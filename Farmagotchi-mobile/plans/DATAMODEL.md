# Farmagotchi Data Model Reference

This document outlines the core data models and schemas required for the Farmagotchi backend. It serves as a reference point for building the database schema (e.g., PostgreSQL, MongoDB) and API responses.

## Core Entities

### 1. User
Represents the farmer/account owner.
```typescript
interface User {
  id: string; // UUID
  email: string;
  name: string;
  location: string; // e.g., "Los Baños"
  createdAt: Date;
  updatedAt: Date;
}
```

### 2. Plot
Represents a specific physical field or crop area. A user can have multiple plots.
```typescript
interface Plot {
  id: string; // UUID
  userId: string; // Foreign Key -> User
  name: string; // e.g., "North Cornfield"
  cropType: string; // e.g., "Corn", "Tomatoes"
  sizeArea?: number; // Optional: Size of the field
  isActive: boolean;
  createdAt: Date;
  updatedAt: Date;
}
```

### 3. Plant (Pet)
The digital pet associated with a specific plot. There is a 1-to-1 relationship between a Plot and a Plant.
```typescript
interface Plant {
  id: string; // UUID
  plotId: string; // Foreign Key -> Plot (Unique constraint)
  name: string; // User-given name for the pet (e.g., "Mango tree")
  petImage: string; // Image filename or key (e.g., "tree.png")
  happiness: number; // Score from 0-100
  createdAt: Date;
  updatedAt: Date;
}
```

### 4. Task
A daily or recurring activity assigned to a specific plot.
```typescript
interface Task {
  id: string; // UUID
  plotId: string; // Foreign Key -> Plot
  title: string; // e.g., "Water the mango tree"
  isCompleted: boolean;
  category: 'daily' | 'weekly' | 'miscellaneous';
  happinessReward: number; // Amount of happiness gained upon completion
  lastCompletedAt?: Date; // To track daily resets
  createdAt: Date;
}
```

### 5. Scan (Crop AI Analysis)
Records of user-uploaded images and the resulting AI analysis for a plot.
```typescript
interface Scan {
  id: string; // UUID
  plotId: string; // Foreign Key -> Plot
  imageUrl: string; // URL to the stored image (S3, Cloud Storage)
  healthScore: number; // 0-100 overall health rating
  anomalies: string[]; // List of detected issues (e.g., ["Pest damage", "Nutrient deficiency"])
  tips: string[]; // Actionable advice generated by the AI
  happinessImpact: number; // The +/- change to the pet's happiness based on this scan
  createdAt: number; // Timestamp (Date.now())
}
```

### 6. Transaction (Finance)
Financial records (income or expense) tied to a specific plot.
```typescript
interface Transaction {
  id: string; // UUID
  plotId: string; // Foreign Key -> Plot
  name: string; // e.g., "Rice", "Fertilizer"
  cost: number; // The amount
  type: 'income' | 'expense';
  createdAt: Date;
}
```

### 7. Chat Message
The history of conversations between the user and their Plant regarding a specific plot.
```typescript
interface ChatMessage {
  id: string; // UUID
  plotId: string; // Foreign Key -> Plot
  role: 'user' | 'assistant';
  content: string; // The text content of the message
  timestamp: number; // Date.now()
}
```
**Note on Persistence:** The system does not support viewing older "sessions." A "Reset Chat" action effectively clears the current message stack for the user.

### 8. Weather
Current weather state for a user's location.
```typescript
interface Weather {
  location: string;
  currentTemp: number;
  highTemp: number;
  lowTemp: number;
  condition: 'cloud' | 'sun' | 'cloud-rain' | 'cloud-lightning' | 'cloud-snow' | 'wind' | 'cloud-drizzle';
}
```

## Business Logic & Context Injection

### AI Chat Context Injection
When a user sends a message, the backend SHOULD NOT just send the raw message to the LLM. It must inject a "Context Snapshot" to make the Plant feel "alive" and aware. The prompt should include:
- **Plant State:** Current name (e.g., "Mango Tree") and Happiness level.
- **Environmental State:** Current weather condition and temperature at the plot's location.
- **Task State:** How many daily tasks are completed vs. pending.
- **Financial State:** Recent net profit/loss status.
- **Scan History:** Results of the most recent crop health analysis.

### Happiness Impact Logic
The backend (or state coordinator) should apply these modifiers to the `Plant.happiness` value:
- **Task Completion:** `+happinessReward` (Base) + `random(0, 5)` (Bonus).
- **Financials:** Income (`+2`), Expense (`-2`).
- **Chat Engagement:** `random(2, 8)` happiness gain when the user asks questions (detected by `?` or intent analysis), with a ~60% probability to prevent spamming rewards.
- **Scans:** AI-generated `happinessImpact` based on the detected crop health.

## Key Relationships
- **User -> Plots:** 1-to-Many.
- **Plot -> Plant:** 1-to-1. Deleting a plot should archive or delete the Plant.
- **Plot -> Tasks, Scans, Transactions, ChatMessages:** 1-to-Many. These are scoped strictly to the `plotId`.
- **Daily Resets:** The backend needs to reset `isCompleted` on `Tasks` where `category` is `daily` at 00:00 local time.
- **Happiness Bounds:** The `happiness` value must always be clamped between `0` and `100`.
